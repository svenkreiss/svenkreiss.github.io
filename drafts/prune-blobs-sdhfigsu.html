Prune Blobs

// under algorithms distributed systems in Tech    // share on Twitter

Discussions of consistent data stores include the Spanner paper with TrueTime (Brewer, 2017), consensus algorithms like Paxos (Lamport and et al., 2001) and Raft (Ongaro and Ousterhout, 2014).

In the given system, there is no eventual consistency. Updates are non-atomic and do not roll back on failure. A convention on the order of updates during blob inserts and blob deletions guarantees that the blob reference count is always equal or higher the true reference count.

The implemented blob store pruning is based on a social assumption: if the refcount today is the same as the refcount yesterday and the same as last week, then that is the correct refcount. This implies that nobody can make a transaction that would affect the count at all three times. The implemented solution produces new refcounts in an efficient manner and checks for consistency across multiple runs.

Algorithmic approach: Min Count Sketch

First idea: Min Count Sketch (Cormode and Muthukrishnan, 2005).

Scaling is bad for large \(\lVert\vec{a}\rVert_1\) which is the number of all references (not even blobs, but references to blobs). A min count sketch guarantee is \(\hat{a}_i \leq a_i + \epsilon \lVert\vec{a}\rVert_1\). This implies that \(\epsilon\) must be very small if \(a_i\) should be within \(\mathcal{O}(1)\) of \(\hat{a}_i\) to distinguish between zero and single references to a blob. Therefore, a min count sketch with good enough precision would consume too much memory.

Systems approach: Materialized Views and Parallel Requests

Define a run_id to be able to form consensus between multiple runs. Per shard, create a materialized view of reference counts for blobs. Two strategies are implemented:

  1. Single query: using GROUP BY on a large UNION.
  2. Incremental: table-by-table queries that update counts using INSERT INTO in combination with ON DUPLICATE KEY UPDATE.

Interface

The command line interface provides two executables, refcount-shard and refcount-global. Both support --help, for example:

usage: refcount-shard [-h] [--db-url DB_URL] [--run-id RUN_ID]

optional arguments:
  -h, --help       show this help message and exit
  --db-url DB_URL  database url
  --run-id RUN_ID  run identifier

Database urls are provided with the command line flag --db-url and are of the form mysql+mysqldb://root@localhost/ProtonMailShard which is the default for refcount-shard.

Conclusions

  • Why count at all? Drop the global refcount table and prune the datastore directly? Dropping the count would simplify this check and improve the performance of frontend services.
