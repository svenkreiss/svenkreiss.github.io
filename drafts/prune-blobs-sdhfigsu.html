Prune Blobs

// under algorithms distributed systems in Tech    // share on Twitter

In the given system, there is no eventual consistency. Updates are non-atomic and do not roll back on failure. A convention on the order of updates during blob inserts and blob deletions guarantees that the blob reference count is always equal or higher the true reference count.

The implemented blob store pruning is based on a social assumption: if the refcount today is the same as the refcount yesterday and the same as last week, then that is the correct refcount. This implies that nobody can make a transaction that would affect the count at all three times. The implemented solution produces new refcounts in an efficient manner and checks for consistency across multiple runs.

This solution does not address how to update the blob store once an inconsistency in the reference count is identified. Even an atomic update of a true inconsistency can make the system inconsistent if another change is happening from another service at that moment.

Algorithmic approach: Count-Min Sketch

First idea: Count-Min Sketch (Cormode and Muthukrishnan, 2005).

Scaling is bad for large \(\lVert\vec{a}\rVert_1\) which is the number of all references (not even blobs, but references to blobs) or “stream size”. A Count-Min sketch guarantee is \(\hat{a}_i \leq a_i + \epsilon \lVert\vec{a}\rVert_1\). This implies that \(\epsilon\) must be very small if \(a_i\) should be within \(\mathcal{O}(1)\) of \(\hat{a}_i\) to distinguish between zero and single references to a blob. Although it is known that the observed precision in practical applications is better than the theoretical bound, a Count-Min sketch is unlikely to be good enough with reasonable memory sizes for the sketch.

Systems approach: Materialized Views and Parallel Requests

Define a run_id to be able to form consensus between multiple runs. Per shard, create a materialized view of reference counts for blobs in the refcount_{runid} table. Two strategies are implemented:

  1. Single query: using GROUP BY on a large UNION.
  2. Incremental: table-by-table queries that update counts using INSERT INTO in combination with ON DUPLICATE KEY UPDATE.

These queries can be run in parallel on all shards. The created table contains columns for blob_id, refcount and verified (a boolean which is set to TRUE if this record was accessed for verification).

On the blob store, run a single full scan that returns batches of blob IDs and their reference counts. For every batch, query all shards in parallel with the list of blob ids to obtain the summed reference counts from the shards. Produce a list of inconsistencies for this run. Monitor this list over time to obtain true inconsistencies.

Interface

The command line interface provides two executables, refcount-shard and refcount-global. Both support --help:

usage: refcount-shard [-h] [--db-url DB_URL] [--global-db] [--run-id RUN_ID]

optional arguments:
  -h, --help       show this help message and exit
  --run-id RUN_ID  run identifier (default: 1517785312)
  --db-url DB_URL  database url (default:
                   mysql+mysqldb://root@localhost/ProtonMailShard)
  --global-db      is this the global database (default: False)

usage: refcount-blobstore [-h] [--blobstore-db-url BLOBSTORE_DB_URL]
                          [--global-db-url GLOBAL_DB_URL]
                          [--shard-db-url SHARD_DB_URL [SHARD_DB_URL ...]]
                          [--run-id RUN_ID]

optional arguments:
  -h, --help            show this help message and exit
  --run-id RUN_ID       run identifier (default: 1517785253)
  --blobstore-db-url BLOBSTORE_DB_URL
                        blobstore database url (default:
                        mysql+mysqldb://root@localhost/ProtonMailGlobal)
  --global-db-url GLOBAL_DB_URL
                        global database url (can be same as blobstore)
                        (default:
                        mysql+mysqldb://root@localhost/ProtonMailGlobal)
  --shard-db-url SHARD_DB_URL [SHARD_DB_URL ...]
                        list of shard database urls (default:
                        ['mysql+mysqldb://root@localhost/ProtonMailShard'])

The global-db is the database containing the SentMessage and SentAttachment tables and blobstore-db contains the BlobStorage table. In the given example they are in the same database.

An example session with explicit parameters:

export RUNID=`date +%s`

# preprocess shards individually (can be parallelized)
refcount-shard --run-id ${RUNID} \
  --db-url mysql+mysqldb://root@localhost/ProtonMailShard
refcount-shard --run-id ${RUNID} \
  --db-url mysql+mysqldb://root@localhost/ProtonMailGlobal --global-db

# scan over blob store
refcount-blobstore --run-id ${RUNID} \
    --blobstore-db-url mysql+mysqldb://root@localhost/ProtonMailGlobal \
    --global-db-url mysql+mysqldb://root@localhost/ProtonMailGlobal \
    --shard-db-url mysql+mysqldb://root@localhost/ProtonMailShard

The last command produces a JSON output of inconsistencies of the form {blob_id: (blob ref count, shard ref count)}. This should be collected over multiple runs spread out over time to find repeated inconsistencies.

Conclusions

Tests are written for pytest and code is linted with flake8 with the stricter hacking extension. Development and packaging notes are in the README.md file.

Tests do not modify the given database tables. To test for missing blob entries or wrong reference counts in the blob store, the reader routine is modified to inject the modified data.

There are many opportunities for performance improvements and the proposed structure does allow for trivial parallelizations.

Questions: Eviction with reference counts is particularly good for short-lived data. I would be curious how many blobs of encrypted text are short lived in the Protonmail system. If the majority of the blobs is long lived, would it be possible to drop the global refcount table and apply the above procedure to the CEPH datastore directly? Dropping the count would simplify this check and improve the performance of frontend services because they don’t have to update reference counts anymore.
